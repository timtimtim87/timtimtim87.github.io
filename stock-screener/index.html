AWS Stock Screener

Serverless stock screening and portfolio tracker with Telegram bot interface. Built to solve a real trading problem — cost ~$0/month, does exactly what is needed, nothing more.

The Problem

I had developed a systematic trading strategy that needed two things: a daily ranked list of entry signals from a 1,000-stock watchlist, and a custom view of active positions showing the specific exit metrics my strategy uses. No commercial tool offered the right combination — Bloomberg Terminal is prohibitively expensive, TradingView doesn't support fully custom metrics, and every dollar spent on tools subtracts directly from trading returns.

The solution needed to be cheap to run, accessible from my phone, and flexible enough to evolve with the strategy. Building it on AWS was the right call.

The Solution — Two Flows

The system is built around two independent flows that work together. The collector runs on a schedule each morning, pulling data from external APIs, computing metrics, and writing results to S3. The bot responds to Telegram commands, reading from what the collector wrote. They share state only through S3 — neither knows the other exists at runtime.

Key Design Decisions

Decision · Compute

Lambda over EC2 or ECS. The workload runs once daily plus sporadic bot queries — compute sits idle 23+ hours per day. EC2 would cost $30–50/month for an always-on instance. Lambda costs ~$1–2/month (mostly free tier) and scales automatically. The workload is clearly event-driven; serverless is the obvious fit.

Decision · Storage

S3 CSV over RDS or DynamoDB. RDS costs $20+/month minimum for an idle instance and adds VPC networking overhead. DynamoDB would require designing partition keys before knowing future analysis needs, and makes local data exploration harder. S3 costs ~$0.30/month, works natively with Python/pandas, and allows downloading files for local analysis whenever needed. The data access pattern is append-only writes and occasional full reads — S3 is the right tool.

Decision · Credentials

Parameter Store over environment variables or Secrets Manager. Parameter Store integrates natively with Lambda via IAM, includes KMS encryption, and costs nothing for standard parameters. API keys are fetched at runtime — never hardcoded, never committed to git, never in plaintext environment variables.

Decision · No VPC

Lambda in public mode, no VPC configuration. Lambda accesses S3 and Parameter Store through AWS internal service endpoints without a VPC. Adding a VPC would require a NAT Gateway (~$45/month) or VPC endpoints (~$7/service/month) for internet access. For this workload, VPC deployment adds cost and complexity with no security benefit.

What I'd Do Differently

Honest reflection is part of how this project demonstrates architectural thinking — not just what was built, but what the tradeoffs were.

  • Data validation. When the Polygon subscription lapsed, the Lambda didn't fail — it logged "downloaded data for 0 stocks" and reported success. Silent failures are worse than loud ones. A simple date-freshness check on the ingested data would have caught this immediately.
  • Resource tagging from day one. Running multiple projects in one account without tagging makes cost attribution impossible. Each project should have been in its own account, or at minimum every resource tagged with a project identifier.
  • S3 versioning strategy. Versioning was enabled without a lifecycle policy, causing exponential bucket growth as each daily CSV created a new version. A lifecycle policy to expire non-current versions after 1 day should have been configured at creation.
