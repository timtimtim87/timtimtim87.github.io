SAM Template Structure

The entire system deploys through AWS SAM (Serverless Application Model), which extends CloudFormation with higher-level constructs for serverless resources. A single template.yaml defines both Lambda functions, their IAM roles, EventBridge scheduling, API Gateway configuration, and S3 bucket setup. 

template.yaml defines:
  ├── DataCollectorFunction
  │   ├── Runtime: python3.11
  │   ├── Memory: 1024 MB
  │   ├── Timeout: 300s
  │   ├── Events: EventBridgeRule (cron)
  │   └── Role: DataCollectorRole
  │
  ├── TelegramBotFunction
  │   ├── Runtime: python3.11
  │   ├── Memory: 512 MB
  │   ├── Timeout: 30s
  │   ├── Events: ApiGatewayWebhook
  │   └── Role: TelegramBotRole
  │
  ├── DataBucket (S3)
  │   ├── SSE-S3 encryption
  │   ├── Versioning: Suspended (lifecycle policy cleans non-current)
  │   └── LifecyclePolicy: delete non-current versions after 1 day
  │
  └── WebhookApi (API Gateway)
      └── POST /webhook → TelegramBotFunction

Deploying changes is sam build && sam deploy — SAM packages the Lambda code, uploads it to S3, and updates the CloudFormation stack.

IAM Design

Each Lambda function has its own IAM execution role with the minimum permissions required. The two roles are kept strictly separate — the bot cannot write data, the collector cannot receive API Gateway events.

RolePermissions
DataCollectorRole s3:PutObject on data bucket
ssm:GetParameter on Polygon + Alpaca parameter paths
logs:* on its own log group
TelegramBotRole s3:GetObject on data bucket (read-only)
ssm:GetParameter on Telegram + Alpaca parameter paths
logs:* on its own log group

Resource ARNs are scoped to specific S3 bucket paths and specific Parameter Store parameter names — not wildcard * resources. If either function were compromised, blast radius is limited to that function's explicit permissions.

Network Architecture

Both Lambda functions run in AWS's default managed environment — no VPC configuration. This is a deliberate cost decision.

OptionMonthly costWhy rejected
No VPC (current)$0—
VPC + NAT Gateway~$45Disproportionate for personal tool
VPC + Interface Endpoints~$14+Same — overkill for this workload

Lambda accesses S3 and Parameter Store through AWS internal service endpoints — traffic routes over AWS's backbone without traversing the public internet. External API calls to Polygon, Alpaca, and Telegram use standard HTTPS. For a personal trading tool, this is the correct tradeoff.

When VPC would make sense. If this system handled PII, processed financial transactions, or needed to access resources inside a private network (e.g. a private RDS instance), VPC deployment would be justified. For Lambda reading and writing CSV files and calling public APIs, it adds no security value.

Lambda Configuration

Memory Sizing

The data collector is configured at 1024 MB — generous for the workload, but Lambda billing is GB-seconds, and given one execution per day the cost impact is negligible (well within free tier). The bot function runs at 512 MB; CloudWatch logs show it uses ~170 MB in practice, suggesting headroom exists.

Cold Starts

Cold starts add 3–4 seconds on the first invocation. For a daily data collection job or an on-demand Telegram bot, this is acceptable. A user waiting 5 seconds for their /daily command is a minor inconvenience. Optimising cold starts (Lambda layers, smaller packages, provisioned concurrency) is not worth the added complexity at this usage scale.

Timeouts

Data collector: 300 seconds. This accommodates fetching 1,000 stock prices from an external API with retry logic for rate limiting. Bot function: 30 seconds, sufficient for S3 reads and Alpaca API calls.

EventBridge Scheduling

The data collector triggers daily at 11 AM UTC via an EventBridge cron rule. This is after US market open (9:30 AM ET) to ensure previous day's EOD data is available from Polygon. 

ScheduleExpression: cron(0 11 * * ? *)

EventBridge delivers the event to the Lambda function. If the invocation fails, Lambda has built-in retry logic (2 retries by default for async invocations). A dead-letter queue could capture failed events for later inspection — not implemented in v1 but a natural next step.

Silent failure discovered post-deployment. When the Polygon subscription lapsed, the Lambda function executed successfully, logged "downloaded data for 0 stocks," and reported success. No alarm fired. The fix: validate that ingested data has today's date before writing to S3. A freshness check on the response timestamp would have caught this immediately.

S3 Storage Design

Three CSV files in one S3 bucket serve as the entire data layer:

FileContentsUpdated
russell_1000_drawdown_results.csvFull 1,000-stock universe with custom metricsDaily (overwrite)
daily_top_candidates.csvTop 10 entry signals ranked by strategy metricDaily (overwrite)
portfolio_snapshots.csvActive Alpaca positions with exit metricsDaily (overwrite)

Files are overwritten daily — no accumulation of historical rows. Historical analysis is done locally by downloading the files; S3 is not used as a time-series store.

Versioning Lesson

S3 versioning was initially enabled without a lifecycle policy. Each daily overwrite created a new version while retaining all previous versions — the bucket grew exponentially. Versioning was suspended and a lifecycle policy was added to delete non-current versions after 1 day. This should have been configured at bucket creation.

Parameter Store

Four API credentials are stored as SecureString parameters in AWS Systems Manager Parameter Store: 

/stock-screener/polygon/api-key
/stock-screener/alpaca/api-key
/stock-screener/alpaca/api-secret
/stock-screener/telegram/bot-token

Parameters are encrypted at rest by KMS. Lambda functions retrieve them at runtime using the AWS SDK — the IAM execution role grants ssm:GetParameter on each specific parameter ARN. No credentials are stored in environment variables, Lambda configuration, or source code.

Parameter Store was chosen over Secrets Manager for this use case — standard parameters are free, and the automatic rotation features of Secrets Manager are not needed for API keys rotated manually.

Data Flow

Daily Collection (11 AM UTC)

  1. EventBridge fires cron rule, invokes Data Collector Lambda
  2. Lambda retrieves Polygon and Alpaca API keys from Parameter Store
  3. Fetches EOD price data for 1,000-stock Russell 1000 universe from Polygon
  4. Calculates custom drawdown metrics and entry signal scores
  5. Fetches active positions from Alpaca API
  6. Calculates strategy-specific exit metrics for each position
  7. Writes all three CSV files to S3 (overwrite)
  8. Lambda exits — compute cost ends

On-Demand Query (Telegram /daily)

  1. User sends /daily command in Telegram
  2. Telegram POSTs to API Gateway webhook endpoint
  3. API Gateway invokes Telegram Bot Lambda
  4. Lambda retrieves Telegram bot token and Alpaca keys from Parameter Store
  5. Reads daily_top_candidates.csv and portfolio_snapshots.csv from S3
  6. Formats results into Telegram message (text-based, no charts)
  7. Sends formatted message via Telegram Bot API
  8. Lambda exits

Telegram Bot Interface

Telegram's bot platform was chosen over a web dashboard for several reasons: native mobile push notifications, simple text-based responses are easier to implement than a charting interface, and the bot API is a straightforward webhook model that maps cleanly to API Gateway + Lambda.

The bot initially had multiple commands. In practice, only /daily was ever used — it returns everything relevant in one response. Unnecessary commands were removed. This is an example of building to actual usage, not hypothetical requirements.

Operational Learnings

  • Silent failures are worse than loud failures. A Lambda that completes successfully with empty output is harder to detect than one that throws an exception. Data validation should be a first-class concern, not an afterthought.
  • S3 versioning requires a lifecycle policy. Enabling versioning without expiry rules causes unbounded storage growth on any bucket with regular overwrites.
  • Tag everything from day one. Running multiple projects in one account without tagging makes cost attribution and resource cleanup significantly harder.
  • Match the interface to actual behaviour. The Telegram bot was over-engineered with multiple commands. Real usage collapsed to one. Build for what you actually need, extend when you need more.
  • Right-size the solution. S3 CSV over RDS, Lambda over EC2, Parameter Store over Secrets Manager — each decision was made by asking "what does this workload actually need?" not "what is the most sophisticated option?"
