AI Integration · Serverless · Web App

SUDOKOO

AI-powered web app that photographs a Sudoku puzzle and turns it into an interactive game — using Amazon Bedrock with Claude 3.5 Sonnet for computer vision extraction.

Amazon Bedrock Lambda CloudFront API Gateway S3 Python

The Problem

My girlfriend had a Sudoku puzzle from a newspaper she'd kept for years. She wouldn't let me write on it — she wanted to finish it herself. I wanted to digitalise it so she could play it without marking the original. That very specific, very personal problem became the starting point.

Beyond solving the niche problem, the project was an opportunity to learn how to integrate AI into a real application using AWS infrastructure — specifically Amazon Bedrock for managed model access.

Architecture Overview

The system is a serverless web application that uses a Lambda function to sends a base64-encoded image to Amazon Bedrock, which uses Anthropic's claude to extract and map the sudoku puzzle from the image and parse a structured response back to the application into create a playable puzzle digitally.

The camera scan is one input mode. The app also includes a client-side puzzle generator — new games are generated in the browser with no API call required. The game engine (pencil marks, keyboard navigation, move validation, a timer, and conflict highlighting) runs entirely client-side and works independently of the AWS backend.

Key Design Decisions

Decision · AI Service Selection

Bedrock over Rekognition or a custom model. Rekognition is designed for object detection and facial recognition — extracting a structured 9×9 grid with cell values is not its intended use case. A custom model would require training data and significant upfront investment. Bedrock with Claude accepts a natural language prompt describing the extraction task and returns the result.

Decision · No VPC

Lambda in public mode. Lambda accesses Bedrock and S3 via AWS internal service endpoints without a VPC. Adding VPC configuration would require a NAT Gateway (~$45/month) with no security benefit for this workload.

Decision · CloudFront + S3 with OAC

Private S3 bucket with Origin Access Control. The bucket policy grants read access only to this specific CloudFront distribution — verified by the distribution's unique ARN. Users access files only through CloudFront, never directly from S3. This prevents bypassing the CDN and avoids unexpected S3 data transfer charges.

Decision · WAF removed

WAF deployed then removed. AWS WAF was initially configured for SQL injection, XSS, and DDoS protection. At $8/month for 3 active rules on a single-user portfolio project, the cost was disproportionate. Removed — but the configuration knowledge is there for production deployments where the cost is justified.

Honest Assessment

The AI extraction works approximately 50% of the time. The two main failure modes are grid alignment errors (correct digits placed in adjacent cells) and character recognition errors (digits misread — most commonly 1 read as 7). Both produce an invalid puzzle — duplicate values in a row, column, or 3×3 box — and are caught by the built-in puzzle validation before loading into the game.

For my portfolio use case, 50% is acceptable, i just wanted to learn how to integrate AI to solve this problem. For a production application, the approach would be different: Claude's structured JSON output mode would replace the current free-text parsing, and a validation pipeline would reject and re-request extractions below a confidence threshold.

Manual deployment through the AWS console was a deliberate choice to build deeper understanding of each service's configuration options. The tradeoff was slow iteration — uploading new frontend files and invalidating CloudFront manually every time became tedious. A CI/CD pipeline (GitHub Actions → S3 sync → CloudFront invalidation) would be the first improvement if this project continued.
