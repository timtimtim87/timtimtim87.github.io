CloudFront + S3 Delivery

The static website (HTML, CSS, JavaScript) is hosted in a private S3 bucket and delivered through CloudFront. CloudFront serves as the only access point — the S3 bucket has no public access.

CloudFront provides two key benefits beyond CDN caching:

  1. HTTPS enforcement. CloudFront issues a free ACM certificate and automatically redirects HTTP to HTTPS. This is required for browser camera access (the Sudoku scan feature uses the device camera, which browsers restrict to secure contexts only).
  2. Cost control. S3 charges for data transfer out. If users could access the bucket directly, every file request would incur S3 egress charges. CloudFront caches at edge locations — subsequent requests are served from cache, not from S3, dramatically reducing transfer costs at any meaningful traffic volume.

Origin Access Control (OAC)

Origin Access Control (OAC) is the mechanism that ties the CloudFront distribution to the S3 bucket. CloudFront uses OAC to sign requests to S3, and the bucket policy requires that signature. 

# S3 bucket policy (simplified)
{
  "Principal": {
    "Service": "cloudfront.amazonaws.com"
  },
  "Condition": {
    "StringEquals": {
      "AWS:SourceArn": "arn:aws:cloudfront::ACCOUNT:distribution/DIST_ID"
    }
  }
}

The condition scopes access to this specific distribution ARN — not all CloudFront distributions in the account, not the CloudFront service generally. Even if someone created another CloudFront distribution pointing at this bucket, the bucket policy would reject requests from it. This is a meaningful isolation boundary.

OAC replaces the older Origin Access Identity (OAI) mechanism. OAC supports additional S3 features (SSE-KMS buckets, multi-region access points) and is the current AWS recommendation.

CORS Configuration

The frontend (served from CloudFront's domain) makes API calls to API Gateway (a different domain). Browsers enforce the same-origin policy and block these cross-origin requests by default. CORS headers on the API Gateway response explicitly allow them. 

Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: POST, OPTIONS
Access-Control-Allow-Headers: Content-Type

Access-Control-Allow-Origin: * allows requests from any origin. This is appropriate for a public web application with no authenticated users — there is no session or credential being shared, just a base64 image being sent and a puzzle grid returned. For an application with authenticated users, this would be restricted to specific trusted origins.

The 30-second timeout is driven by Bedrock response latency. Claude 3.5 Sonnet processing a high-resolution image with a detailed extraction prompt takes 4–12 seconds in practice. The timeout provides headroom for occasional slower responses without failing the user request.

Memory at 512 MB handles base64 encoding of a typical phone camera image (3–8 MB JPEG → ~4–11 MB base64) in memory alongside the Python runtime. CloudWatch logs confirm actual usage is well below 512 MB.

API Gateway

A single API Gateway resource handles the puzzle scan endpoint: POST /scan. The request body contains a base64-encoded image and is passed directly to the Lambda function as the event payload.

API Gateway handles SSL termination, request routing, and throttling. The default throttle limits (10,000 requests/second, 5,000 burst) are far beyond anything this application will ever receive — they exist as a safety net against runaway costs if the endpoint were somehow abused.

Binary media type handling. The image is encoded as base64 by the browser before sending — the Lambda function receives a base64 string, not raw binary. This avoids configuring API Gateway binary media types, which adds complexity. The tradeoff is ~33% payload size overhead from base64 encoding, acceptable for images up to ~10 MB.

Network Design

Lambda runs in AWS's managed environment with no VPC configuration. This is identical to the Stock Screener decision — the workload calls public APIs (Bedrock) and accesses AWS-internal services (S3), neither of which requires VPC deployment.

Lambda accessing Bedrock routes over AWS's internal network. Lambda accessing S3 for (future) image storage would also route internally. No traffic to AWS services traverses the public internet.

The key question when evaluating VPC deployment: does this workload need to access private network resources? For SUDOKOO, the answer is no. All resources are either AWS managed services (accessible internally) or public APIs (HTTPS).

Bedrock Integration

Amazon Bedrock provides managed access to foundation models without infrastructure management. The Lambda function calls the Bedrock runtime API with the image and a structured extraction prompt. 

# Lambda invocation (simplified)
response = bedrock_client.invoke_model(
    modelId='anthropic.claude-3-5-sonnet-20241022-v2:0',
    body=json.dumps({
        'anthropic_version': 'bedrock-2023-05-31',
        'max_tokens': 1000,
        'messages': [{
            'role': 'user',
            'content': [
                {
                    'type': 'image',
                    'source': {
                        'type': 'base64',
                        'media_type': 'image/jpeg',
                        'data': base64_image
                    }
                },
                {
                    'type': 'text',
                    'text': 'Extract the 9x9 Sudoku grid...'
                }
            ]
        }]
    })
)

Why Claude 3.5 Sonnet specifically?

Claude 3.5 Sonnet offers the best combination of vision capability and cost efficiency in the Bedrock model catalogue. Claude 3 Opus has slightly better reasoning but costs ~5x more per token — not justified for a grid extraction task. Claude 3 Haiku is cheaper but vision capability is less reliable for structured extraction with ambiguous handwriting.

Failure Modes

The AI extraction has two main failure modes, observed through CloudWatch logs by comparing response output with the original puzzle photos:

  1. Grid alignment errors. Correct digit values placed in adjacent cells — typically off by one row or column. These pass character recognition but fail puzzle validation (duplicate values in row/column/box).
  2. Character recognition errors. Digits misread — most commonly 1 read as 7, or 6 read as 8. These may pass puzzle validation if the misread doesn't create an immediate conflict, producing a subtly wrong puzzle.

The built-in puzzle validity checker (verifying no duplicate values in any row, column, or 3×3 box) catches alignment errors. Character recognition errors are harder to detect automatically without a reference answer.

Overall success rate: approximately 50%. For a single-user personal tool, this is acceptable. Re-scanning with better lighting or a different angle usually resolves failures.

Future Improvements

These are genuine improvements I would make if this project were production-bound — not hypothetical additions included for portfolio completeness.
  • Structured JSON output. Claude now supports structured output mode, which guarantees a specific JSON schema in the response. The current implementation parses free-text output, adding complexity and fragility. Structured output would simplify the Lambda function significantly and improve reliability.
  • CI/CD pipeline. Manual S3 upload + CloudFront invalidation per frontend change was the most operationally painful part of this project. A GitHub Actions workflow (push to main → S3 sync → invalidation) would take 30 minutes to implement and save significant time on any future iteration.
  • Confidence threshold and retry. If Bedrock returns a response with validation errors, automatically retry with a prompt variation before returning failure to the user. One retry would likely push success rate above 70%.
  • Resource tagging. All resources should be tagged from day one for cost attribution and resource management. Lessons learned from the Stock Screener project not applied here — a mistake not to repeat.

IAM Design

The Lambda execution role follows least-privilege with three explicit permissions: 

# IAM inline policy
{
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "bedrock:InvokeModel",
      "Resource": "arn:aws:bedrock:us-east-1::foundation-model/
                   anthropic.claude-3-5-sonnet-20241022-v2:0"
    },
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:us-east-1:ACCOUNT:log-group:
                   /aws/lambda/sudokoo-scan:*"
    }
  ]
}

The Bedrock permission is scoped to the specific model ARN — not all Bedrock models. If a different model is needed in future, it requires an explicit IAM update, not an accidental discovery that the role had broader access than intended.

The CloudWatch Logs permission is scoped to this function's specific log group ARN. The function cannot write to any other log group.

WAF Decision

AWS WAF was deployed initially with three managed rules: SQL injection protection, XSS protection, and the common ruleset (CRS). Cost: $8/month fixed plus per-request charges.

For a single-user portfolio project with no user accounts, no stored data, and no financial transactions, $8/month is disproportionate. WAF was removed.

The decision criteria for reinstatement: authenticated users, stored PII, payment processing, or traffic patterns suggesting active abuse. None of these apply. Documenting the decision and the criteria for reversal is more useful than leaving WAF in place at unjustified cost.

Bedrock pricing for Claude 3.5 Sonnet: $3/1M input tokens + $15/1M output tokens. A typical image scan uses ~1,500 input tokens (image + prompt) and ~200 output tokens. Cost per scan: approximately $0.007. At any personal usage level, this is negligible.

At scale (e.g. 10,000 scans/month): ~$70/month in Bedrock costs. At that point, evaluating a custom fine-tuned model on SageMaker would make sense — higher upfront cost offset by much lower per-inference pricing.

Deployment Process

This project was deliberately deployed through the AWS console rather than IaC. The goal was developing hands-on understanding of each service's configuration options — something that IaC abstracts away.

The tradeoff became clear quickly: manual deployment is fine for one-time setup but painful for iteration. Every frontend change required:

  1. Upload new files to S3 via console or CLI
  2. Manually create a CloudFront invalidation (/*)
  3. Wait 60–90 seconds for invalidation to propagate

A GitHub Actions workflow handling this automatically would be the first change made if development continued. The console-first approach achieved its learning goal; any future project starts with IaC from day one.
