AWS Stock Screener

Serverless stock screener and portfolio tracker with Telegram bot interface. Built to solve a real trading problem — cost ~$0/month, does exactly what is needed, nothing more.

The Problem

I had developed a systematic trading strategy that needed two things: a daily ranked list of entry signals from a ~1,000-stock watchlist, and a custom view of active positions showing strategy-specific exit metrics — drawdown from peak, recovery rate, and signal aging. No commercial tool offered the right combination. Bloomberg Terminal is prohibitively expensive, TradingView doesn't support fully custom metrics, and every dollar spent on tooling subtracts directly from trading returns.

The solution needed to be cheap to run, accessible from my phone, and flexible enough to evolve with the strategy. Building it on AWS was the right call.

What It Does

Every morning before market open, a Lambda function ingests end-of-day price data for ~1,000 stocks, computes ranking metrics in Python, and writes three CSV files to S3 — a full universe results file, a top-candidates shortlist, and a portfolio snapshot. When I send /daily from Telegram, a second Lambda reads those files, fetches live position data from my broker API, and sends back a formatted message with ranked candidates and current position metrics. The whole thing runs in under 30 seconds and costs effectively nothing.

Outcome
~1,000 stocks screened and ranked daily before market open
Ranked entry candidates delivered to Telegram on demand
Custom exit metrics unavailable in off-the-shelf tools
Replaced a slow, inconsistent manual workflow
Runs at ~$0–2/month (mostly within free tier)

Architecture Overview

Two Lambda functions, two trigger mechanisms, one storage layer, one credential store. The collector and the bot share state only through S3 — neither knows the other exists at runtime. This decoupling keeps failure domains separate: a bad collection run doesn't affect the bot's ability to serve the last good data, and a bot failure doesn't disrupt collection. It also makes the outputs inspectable locally by just downloading the CSV files from S3.

Key Design Decisions

Decision · Compute

Lambda over EC2 or ECS. The workload runs once daily plus sporadic bot queries — compute sits idle 23+ hours per day. EC2 would cost $30–50/month for an always-on instance. Lambda costs ~$1–2/month (mostly free tier) and scales automatically. The workload is infrequent, bursty, and event-driven — serverless is a strong fit on both cost and operational overhead.

Decision · Storage

S3-based file storage over RDS or DynamoDB. RDS costs $20+/month minimum for an idle instance and adds VPC networking overhead. DynamoDB would require designing partition keys before knowing future analysis needs, and makes local data exploration harder. S3 costs ~$0.30/month, works natively with Python/pandas, and allows downloading files for local analysis whenever needed. The data access pattern is append-only writes and occasional full reads — S3 is the right tool.

Decision · Credentials

Parameter Store over environment variables or Secrets Manager. Parameter Store integrates natively with Lambda via IAM, includes KMS encryption, and costs nothing for standard parameters. API keys are fetched at runtime — never hardcoded, never committed to git, never in plaintext environment variables. Secrets Manager would be the right choice if rotation were needed; for static API keys, Parameter Store is the simpler and cheaper option.

Decision · No VPC

Lambda in public mode, no VPC configuration. Lambda accesses S3 and Parameter Store through AWS internal service endpoints without a VPC. Adding a VPC would require a NAT Gateway (~$45/month) or VPC endpoints (~$7/service/month) for internet access. For this workload, VPC deployment adds cost and networking complexity without materially improving the security posture.

Where This Design Breaks Down

This architecture is well-suited to scheduled batch ingestion, low-frequency query traffic, and file-oriented analytics. It would stop being the right fit if:

  • Query volume increased substantially — reading full CSVs on every bot request doesn't scale. A metadata index or DynamoDB layer for keyed lookups would be the next step.
  • Latency became critical — cold Lambda starts and S3 reads add up. A warm cache layer would help.
  • Multiple users needed access — the current design assumes a single user. Multi-user access would require per-user data isolation and an auth layer.
  • The data model grew more complex — CSV works well for flat tabular data. Parquet + Athena would be a better fit for larger datasets with ad hoc query needs.

For the actual use case — one user, one strategy, daily batch — none of these constraints are relevant, and the simplicity of the design is the point.

What I'd Do Differently

Honest reflection is part of how this project demonstrates architectural thinking — not just what was built, but what the tradeoffs were.

  • Data validation. When the Polygon subscription lapsed, the Lambda didn't fail — it logged "downloaded data for 0 stocks" and reported success. Silent failures are worse than loud ones. A simple date-freshness check on the ingested data would have caught this immediately.
  • Resource tagging from day one. Running multiple projects in one account without tagging makes cost attribution impossible. Each project should have been in its own account, or at minimum every resource tagged with a project identifier.
  • S3 versioning strategy. Versioning was enabled without a lifecycle policy, causing exponential bucket growth as each daily CSV created a new version. A lifecycle policy to expire non-current versions after 1 day should have been configured at creation.
  • Single data provider. Using both Polygon and Alpaca added unnecessary complexity. Alpaca supports EOD price data — consolidating to one provider would simplify the architecture and reduce credential surface area.
