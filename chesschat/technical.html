VPC Architecture

The network is a three-tier VPC across three Availability Zones in us-east-1 (CIDR 10.0.0.0/16). Each tier has a distinct trust level enforced by both route tables and security groups — defense in depth, not reliance on a single control.

Three-tier subnet layout, AZ distribution, route tables, endpoints — to be added

TierSubnetsHostsInternet

Private App 10.0.11–13.0/24 ECS Fargate tasks Outbound only via NAT Gateway
Private Data 10.0.21–23.0/24 ElastiCache Redis None — no internet route

Why Three Tiers?

A common simplification is two tiers (public + private). The third tier adds explicit network-level enforcement that Redis is unreachable from anywhere except the app tier. Security groups already enforce this, but route tables in the data subnets have no 0.0.0.0/0 entry — there is no internet path even if a security group rule were misconfigured. Two independent controls are better than one.

NAT Gateway Strategy

A single NAT Gateway handles outbound internet from the private app subnets. This introduces a single-AZ egress dependency — accepted for the portfolio phase. The Terraform VPC module accepts nat_gateway_mode = "single" | "per_az" so switching to per-AZ NAT (and the ~$32/month additional cost) requires one variable change. See ADR-0001 ↗.

Security Groups

Security groups enforce a strict one-directional trust chain. Each layer accepts traffic only from the layer directly above it — no lateral movement is possible. 

Internet (0.0.0.0/0)
  │  443 / 80
  ▼
ALB Security Group
  │  8080 (app container port)
  ▼
Fargate Task Security Group
  │  6379 (Redis)
  ▼
Redis Security Group

The Redis security group source is the Fargate task security group ID, not a CIDR block. This means any new Fargate task automatically inherits access when it joins the security group, and no IP management is required as tasks scale.

A separate VPC Endpoint security group allows port 443 from the VPC CIDR (10.0.0.0/16) to reach AWS service endpoints — ECR, CloudWatch Logs, Secrets Manager.

VPC Endpoints

AWS API calls from Fargate tasks route through VPC Endpoints rather than the NAT Gateway. This eliminates NAT data processing charges on ECR image pulls, CloudWatch log writes, and Secrets Manager reads — and keeps traffic inside the AWS network.

EndpointTypeCostPurpose
S3GatewayFreeECR image layer storage, static assets
DynamoDBGatewayFreeDatabase access from app tier
ECR APIInterface~$7/moContainer image metadata
ECR DockerInterface~$7/moContainer image pull
CloudWatch LogsInterface~$7/moApplication log delivery
Secrets ManagerInterface~$7/moRedis auth token at task startup

Gateway endpoints are free and route traffic through the AWS backbone without leaving the VPC. Interface endpoints cost ~$7/month each but eliminate NAT data charges that would exceed this on any meaningful ECR pull volume.

Terraform Structure

All infrastructure is defined in Terraform with a modular structure — one module per AWS service boundary. The root main.tf wires modules together, passing outputs as inputs (e.g. VPC subnet IDs feed into ECS and ElastiCache). 

terraform/
├── modules/
│   ├── vpc/          # VPC, subnets, IGW, NAT, route tables, endpoints
│   ├── ecs/          # Cluster identity, IAM roles, security groups
│   ├── ecs_compute/  # Task definitions, ECS service, autoscaling
│   ├── alb/          # ALB, target groups, listeners, ACM cert
│   ├── dynamodb/     # All DynamoDB tables and GSIs
│   ├── elasticache/  # Redis replication group, subnet group, param group
│   ├── cognito/      # User pool, app client, Google IdP
│   ├── route53/      # Hosted zone, alias records, ACM validation
│   └── monitoring/   # CloudWatch alarms, dashboards, SNS, budgets
├── environments/dev/
│   └── terraform.tfvars
├── main.tf           # Module wiring
├── backend.tf        # S3 remote state config
└── versions.tf       # Provider version pins
ECS split into two modules. The ECS identity module (ecs/) creates the cluster, IAM roles, and security groups. The ECS compute module (ecs_compute/) creates task definitions and the service. This split breaks a circular dependency: the Redis security group needs to allow the Fargate task security group as a source, but the task definition needs the Redis endpoint. Separating the resources removes the cycle.

Module Outputs as Inputs

Modules expose typed outputs that other modules consume directly. For example: 

# In main.tf
module "elasticache" {
  source             = "./modules/elasticache"
  private_data_subnet_ids = module.vpc.private_data_subnet_ids
  fargate_sg_id      = module.ecs.fargate_task_sg_id
}

module "ecs_compute" {
  source         = "./modules/ecs_compute"
  redis_endpoint = module.elasticache.primary_endpoint
  redis_secret_arn = module.elasticache.auth_token_secret_arn
}

This wiring means Terraform's dependency graph is explicit — it knows to create ElastiCache before the ECS service, and to destroy the ECS service before ElastiCache.

Remote State

Terraform state is stored in S3 with versioning enabled and file-based locking (use_lockfile = true). State versioning means any previous infrastructure state can be restored from S3 version history if needed. 

# backend.tf
terraform {
  backend "s3" {
    bucket       = "chesschat-tfstate-723580627470-us-east-1"
    key          = "dev/terraform.tfstate"
    region       = "us-east-1"
    use_lockfile = true
  }
}

The state bucket name includes the AWS account ID to guarantee global uniqueness without a random suffix that would need to be tracked separately.

CI/CD Pipeline

GitHub Actions handles continuous deployment. Authentication to AWS uses OIDC (OpenID Connect) — no long-lived access keys are stored as GitHub secrets. Instead, GitHub's OIDC provider is trusted by an IAM role, and the workflow assumes that role for the duration of the pipeline run.

CI/CD Flow Diagram

GitHub Actions → OIDC → AWS → ECR push → ECS deploy — to be added

Pipeline stages on push to main:

  1. Run frontend tests (vitest)
  2. Build Docker image
  3. Authenticate to AWS via OIDC
  4. Push image to ECR with commit SHA tag
  5. Update ECS task definition to new image tag
  6. Deploy new ECS service revision (rolling update)
  7. Static site assets synced to S3 + CloudFront invalidation

The static site deploy (Step 7) and the ECS deploy (Steps 4–6) are independent pipelines — a frontend-only change does not trigger an ECS deployment.

Redis vs DynamoDB — The Split

The platform uses two data stores with deliberately separate responsibilities. The guiding principle: Redis holds what is needed right now; DynamoDB holds what needs to be remembered.

DataStoreReason
Active room metadataRedis (TTL)Ephemeral, fast access, irrelevant after session
In-progress game stateRedis (TTL)Sub-ms read/write for move validation and clocks
WebSocket connection mapRedisShared across Fargate tasks for scalable routing
Completed game recordsDynamoDBPermanent, queryable by player and date
User profiles and statsDynamoDBPermanent, low write frequency
Social graphDynamoDBPermanent, structured relationships

Importantly, DynamoDB receives exactly one write per completed game, regardless of how many moves were played. All mid-game state lives in Redis. This keeps DynamoDB costs proportional to games completed, not game activity.

DynamoDB Schema

The schema uses a multi-table design — one table per domain — rather than the popular single-table pattern. IAM permissions, CloudWatch metrics, and PITR settings are all cleaner per-table. The social domain was also added after initial design; retrofitting single-table key namespacing would have been disruptive.

Entity Relationship Diagram

DynamoDB table relationships and GSI structure — to be added

Users Table

PK: user_id (S)   ← Cognito sub (immutable UUID)

Attributes: username, display_name, email,
            chess_com_username, chess_com_verified,
            total_games, wins, losses, draws,
            created_at, last_login_at

GSI: username-index
  PK: username (S)
Why user_id as PK, not username? Usernames can change. The chess.com OAuth integration includes a displacement mechanic where a verified chess.com holder can claim a username held by an unverified user. Using an immutable Cognito sub means a username change is a single UpdateItem, not a data migration.

Games Table

PK: game_id (S)   ← UUID
SK: ended_at (N)  ← Unix timestamp (ms)

Attributes: white_player_id, black_player_id,
            winner, result, total_moves, duration_seconds,
            moves[], fen_history[], pgn, room_code,
            time_white_remaining, time_black_remaining

GSI-1: white-player-index   PK: white_player_id, SK: ended_at
GSI-2: black-player-index   PK: black_player_id, SK: ended_at
Why two GSIs? A player alternates colours across games. To query their full history you need games where they were white AND games where they were black. DynamoDB GSI partition keys map to a single attribute — two GSIs with two parallel queries (merged in the application) is the idiomatic solution. A single GSI cannot serve both lookup directions.

Social Tables

Four additional tables support the social layer:

Friendships are written in both directions simultaneously (A→B and B→A), so listing a user's friends is a single Query PK=user_id with no GSI required. The Challenges table mirrors the two-GSI pattern from Games — a challenge involves two parties and must be visible to both.

Redis Schema

Redis stores ephemeral session state. All keys have a TTL and are treated as disposable — Redis state loss is survivable; DynamoDB loss is not. 

room:{room_code}          Hash, TTL 60min
  status                  waiting | active | teardown
  chime_meeting_id
  chime_meeting_data      JSON blob
  participant:{uid}:*     Per-user connection fields
  active_game             JSON blob (null if no game running)
  games_played            JSON array of completed summaries

ws:{connection_id}        Hash, TTL 60min
  user_id
  room_code
  connected_at

WebSocket connection state lives in Redis (not in Fargate task memory) so that any task can look up any connection. This is the foundation for distributed WebSocket routing when the single-task guardrail is lifted.

Data Flow

Data Flow Diagram

Request path from browser through CloudFront / ALB to Fargate, Redis, DynamoDB — to be added

Room Creation

  1. User A clicks Create Room → WebSocket message to Fargate
  2. Backend generates 8-char alphanumeric room code
  3. Room written to Redis with 60-minute TTL
  4. Room code returned to User A's browser
  5. User B enters code → backend validates room exists
  6. Backend creates Chime SDK meeting, generates attendee tokens for both
  7. Meeting credentials pushed to both clients via WebSocket
  8. Chime SDK initialises on both frontends — video active

Chess Move

  1. Player drags piece — chess.js validates move locally (UX speed)
  2. Move sent to backend via WebSocket: { move: "e2e4" }
  3. Backend validates server-side with Stockfish (security)
  4. Board FEN, move list, and clocks updated in Redis
  5. Move broadcast to both clients: move_made event with audio metadata
  6. Both frontends update board, clocks, and play sound cue

Game Completion

  1. End condition detected (checkmate / resignation / timeout)
  2. Game metadata calculated (duration, time remaining, move count)
  3. Game record written to DynamoDB — single PutItem
  4. User stats updated — UpdateItem on Users table
  5. Active game cleared from Redis room state
  6. game_ended event broadcast to both clients

Session Lifecycle

Rooms follow phone-call semantics. The lifecycle is deterministic with no ambiguous intermediate states. 

Room created (User A)
  │  status: waiting
  ▼
User B joins
  │  status: active — Chime meeting created
  ▼
Games played (unlimited sequential games)
  │  Each game: Redis active_game → DynamoDB on completion
  ▼
Player disconnects
  │  12-second reconnect grace
  │  If grace expires → teardown begins
  ▼
Both players leave (or grace expires)
  │  Chime meeting deleted
  │  Active game (if any) written to DynamoDB as abandonment
  │  Redis room key deleted
  ▼
Room consumed — code is single-use

The tombstone (record that the room existed and was torn down) is written atomically with the Redis deletion to prevent race conditions on room code reuse.

WebSocket Routing

WebSocket connections require session affinity — a client must reach the same Fargate task to maintain its connection. The ALB target group has sticky sessions enabled (lb_cookie, 24-hour duration).

Current scaling constraint. WebSocket routing is single-task until a Redis pub/sub message relay is implemented. At that point, any task can relay messages to any other task and sticky sessions become a preference, not a requirement. This constraint is documented and the upgrade path is clear.

Connection state is stored in Redis (not task memory), so the foundation for distributed routing is already in place. Implementing the pub/sub relay is the remaining step.

Video Infrastructure

Current: Amazon Chime SDK

Chime SDK provides a managed WebRTC implementation. The server creates a meeting and issues attendee tokens; the client SDK handles negotiation, STUN/TURN, and media streams. Cost: $0.0017 per attendee-minute.

A 1-hour session with two players = 120 attendee-minutes = $0.20 per session.

Planned: Native WebRTC + Coturn

The migration replaces Chime SDK with browser-native WebRTC and a self-hosted Coturn TURN server. Cost becomes a flat EC2 instance fee rather than per-minute billing.

Chime SDKWebRTC + Coturn
Cost model$0.0017/attendee-minFlat EC2 (~$15–30/mo)
Break-even—~150 hours/month of video
ControlManaged (limited)Full (codec, bitrate, signalling)
TURN requiredManaged by ChimeSelf-hosted (~15% of sessions)
SFU neededNo (1-on-1)No (1-on-1)

Migration sequence: keep Chime now → swap to WebRTC + single Coturn instance pre-launch → add redundancy post-launch → add regional instances based on user geography.

IAM Role Separation

ECS uses two distinct IAM roles per task:

  • Task Execution Role — used by the ECS agent to pull the container image from ECR, write logs to CloudWatch, and fetch secrets from Secrets Manager at startup. The application code never uses this role.
  • Task Role — used by the application at runtime. Grants access to specific DynamoDB table ARNs, Chime SDK meeting operations, and ElastiCache describe operations. No wildcard resources.

Logging

Application logs write to CloudWatch Logs via the awslogs driver in the ECS task definition. Log group /ecs/chesschat/application, 30-day retention. VPC Flow Logs write all traffic to a separate log group, 7-day retention.
